Курс: Шаблоны проектирования приложений

Тема: Модуль 11 Диаграмма классов, Диаграмма компонентов
Баллы: 
Цель работы:
Разработать UML-диаграмму классов для сложной системы управления университетом, учитывая множество сущностей и их взаимодействия.
Условия:
1.	Сущности системы:
o	Университет
o	Факультет
o	Кафедра
o	Преподаватель
o	Студент
o	Курс (учебная дисциплина)
o	Расписание
o	Экзамен
o	Оценка
o	Администратор (системы)
2.	Функциональность системы:
o	Студенты регистрируются на курсы и сдают экзамены.
o	Преподаватели ведут курсы, выставляют оценки и управляют расписанием.
o	Администратор управляет структурами (факультетами, кафедрами, курсами) и пользователями (преподавателями, студентами).
o	Кафедры привязаны к факультетам.
o	Каждый курс закреплен за одной кафедрой.
o	У каждого преподавателя есть список курсов.
o	Расписание связывает курсы, студентов и преподавателей.
3.	Требования к сущностям:
o	Каждая сущность должна иметь атрибуты и методы.
o	Учесть наследование, если сущности имеют общие свойства.
o	Отразить связи (ассоциации, агрегации, композиции) между классами.

Пример структуры:
1. Класс Университет
	Атрибуты: название, адрес, списокФакультетов.
	Методы: добавитьФакультет(), удалитьФакультет().
2. Класс Факультет
	Атрибуты: название, списокКафедр.
	Методы: добавитьКафедру(), удалитьКафедру().
3. Класс Кафедра
	Атрибуты: название, списокПреподавателей, списокКурсов.
	Методы: добавитьПреподавателя(), удалитьПреподавателя().
4. Класс Студент (наследуется от Пользователь)
	Атрибуты: номерЗачетки, списокКурсов.
	Методы: зарегистрироватьсяНаКурс(), просмотретьОценки().
5. Класс Преподаватель (наследуется от Пользователь)
	Атрибуты: списокcКурсов, должность.
	Методы: провестиЗанятие(), выставитьОценку().
6. Класс Курс
	Атрибуты: название, кафедра, списокСтудентов, расписание.
	Методы: добавитьСтудента(), удалитьСтудента().
7. Класс Расписание
	Атрибуты: дата, время, аудитория.
	Методы: создатьРасписание(), изменитьРасписание().
8. Класс Экзамен
	Атрибуты: курс, дата, списокОценок.
	Методы: назначитьЭкзамен(), выставитьОценку().

Отношения:
	Университет → Факультет: (один университет содержит множество факультетов).
	Факультет → Кафедра: (один факультет содержит множество кафедр).
	Кафедра → Преподаватель.
	Курс → Преподаватель: (один курс может вестись несколькими преподавателями).
	Курс → Студент: (один курс может содержать много студентов).
	Расписание → Курс:
	Экзамен → Оценка:

Задание:
1.	Опишите все классы, их атрибуты и методы.
2.	Определите связи между классами (укажите типы отношений: ассоциация, агрегация, композиция, наследование).
3.	Постройте UML-диаграмму классов с использованием программного обеспечения (StarUML, Draw.io, Visio) или вручную.
4.	Включите в диаграмму хотя бы 10 классов и правильно распределите между ними связи.

Дополнительное задание:

	Отразите их взаимодействие с уже созданными классами.




Тема: Система онлайн-банкинга
Цель работы:
Разработать UML-диаграмму компонентов для системы онлайн-банкинга.
Описание системы:
Система онлайн-банкинга должна предоставлять клиентам доступ к следующим функциям:
	Управление счетами (просмотр баланса, история операций).
	Переводы средств (внутренние и международные).
	Оплата услуг (коммунальные платежи, мобильная связь).
	Управление картами (выпуск, блокировка).
	Авторизация и безопасность (двухфакторная аутентификация, шифрование).
	Поддержка пользователей (чат и заявки).
Требования:
1.	Основные компоненты системы:
o	Frontend: Веб-приложение и мобильное приложение.
o	Backend: Серверная логика, включая обработку запросов клиентов.
o	База данных: Хранение информации о клиентах, счетах, транзакциях.
o	Система авторизации: Управление пользователями и их правами.
o	Платежный шлюз: Обработка переводов и оплат.
o	Система уведомлений: Уведомления по SMS и email.
o	Модуль аналитики: Формирование отчетов и обработка статистики.
o	Система мониторинга: Отслеживание состояния системы.
2.	Связи между компонентами:
o	Frontend взаимодействует с Backend через REST API или GraphQL.
o	Backend обращается к Базе данных для получения и обновления данных.
o	Платежный шлюз обеспечивает связь с внешними банками и провайдерами услуг.
o	Система авторизации проверяет права доступа для каждого действия.
o	Система уведомлений отправляет клиентам SMS и email.
o	Модуль аналитики обрабатывает данные из базы и генерирует отчеты.
o	Система мониторинга проверяет доступность и производительность всех компонентов.
3.	Функциональные требования:
o	Поддержка безопасности (например, шифрование данных, защита API).
o	Высокая производительность и масштабируемость (обеспечение работы с тысячами пользователей одновременно).

Задание:
1.	Определите основные компоненты системы и их ответственность.
2.	Укажите интерфейсы для взаимодействия между компонентами (например, REST API для Frontend и Backend).
3.	Постройте диаграмму компонентов, отразив:
o	Компоненты системы.
o	Взаимодействие между ними через интерфейсы.
o	Внешние системы (например, провайдеры платежей).
4.	Используйте любой UML-инструмент для построения диаграммы (StarUML, Lucidchart, Draw.io, Visio).

Пример структуры компонентов:
	Frontend
o	Интерфейс: REST API
o	Задачи: Обеспечение взаимодействия с пользователем.
	Backend
o	Интерфейс: Database, Auth Service, Payment Gateway
o	Задачи: Обработка логики приложения, маршрутизация запросов.
	Database
o	Интерфейс: SQL
o	Задачи: Хранение данных о клиентах, транзакциях, счетах.
	Payment Gateway
o	Интерфейс: External API
o	Задачи: Обеспечение платежей и переводов.
	Auth Service
o	Интерфейс: Token Authentication
o	Задачи: Авторизация и аутентификация пользователей.
